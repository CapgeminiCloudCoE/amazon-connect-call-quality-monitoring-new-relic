Setup

If you deployed AmazonConnectMonitoringSolution — version 1.0.2 (1 October 2020), you will need following the following steps to manually configure High cardinality Anomaly detection.

If you deployed AmazonConnectMonitoringSolution — version 2.0.1 (25th November 2020), you can skip this section and go straight to the user guide.

  1. Follow the steps here to upgrade your ES cluster to at least version 7.9 with the steps here:

https://docs.aws.amazon.com/elasticsearch-service/latest/developerguide/es-version-migration.html

  1. Navigate to anomaly detection in Kibana

  1. Click Create a detector

  2. Fill in the appropriate fields

Parameter Value Description
Name audio-quality The name of your detector.
Description The audio quality detector based on the softphone-stream-stats index Optional description
Index softphonestreamstats- The index monitored by the detector.
Timestamp doc.timestamp The timestamp attribute used by the detector.
Detector Interval 2 minutes Sets the time interval at which the detector collects data.
Window delay 1 minute This is to tell the detector that the data is not ingested into Elasticsearch in real time but with a certain delay. The value is 1 minute because the index is populated at a 30 second interval
Detector Interval 2 minutes The name of your detector.

  1. Once the detector has been created successfully, click configure model

  2. We can now add features to the detector.

We are adding three features to the detector, using the average() aggregation method.

  • Round trip time
  • Jitter
  • Packet loss

  1. To enable detection at the agent level we can add doc.agent.keyword as the category and window size of 1 interval.

  1. Once we click save and start detector the detector will start initializing.

  2. Once the detector is ready it will be in the “running” state.

User guide

Please wait for the detector to be in the running state before continuing.

  1. Once the detector is ready we can filter based on time range and see a heat map of the anomalies.

  2. We can examine each anomaly by clicking on the tile in the heatmap. In this example we can see an anomaly grade range of 0.1 - 0.85 and confidence of 0.83. These values are based on the anomaly occurrences from the time range which can be looked at individually for further investigation.

  1. We can toggle the “feature breakdown” tab to see how each of the features contributed to the anomaly.

Based on these graphs we can see anomaly values in all three features, round trip time, packet loss and jitter.

Alerting

We can also set up alerting based on anomalies.

  1. Click “set up alerts” to begin.

  1. First we create a monitor

  2. Then we create the trigger. You can customize the level of anomaly grade to be alerted on.

  3. Create the action. In our example we are using an SNS destination. If you don’t have any destinations configured please follow the next step to add a destination.

  4. Add destination.

  5. Once the set up is complete your destination will be alerted base on the anomalies.
